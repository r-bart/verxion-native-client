## Findings

### Major - Route bootstrap lives in `app/_layout.tsx`

Evidence: `app/_layout.tsx` has 26 lines and owns global CSS import, splash
screen prevention, font loading, and `GestureHandlerRootView`. The local rule
says route files should be 3-5 lines and delegate composition to
`presentation/app`.

Risk: the root route becomes a second composition root, making the app-shell
exception harder to reason about.

Recommendation: move the bootstrap body into a `presentation/app/RootLayout` or
similar app-shell component. Leave `app/_layout.tsx` as an import + default
render.

### Major - `presentation/app` imports infrastructure i18n directly

Evidence: `src/presentation/app/AppProvider.tsx` imports
`@/infrastructure/i18n/i18n` as a side effect. The documented app-shell exception
allows `infrastructure/di` bootstrap artifacts, not arbitrary infrastructure
modules.

Risk: the exception broadens informally; future providers may import concrete
infrastructure directly from presentation.

Recommendation: move i18n startup into an infrastructure bootstrap module that
is reached through DI/container initialization, or explicitly document i18n as
part of the app-shell exception.

### Major - Production repositories still serve domain fixtures

Evidence: `src/infrastructure/repositories/HttpTrainingRepository.ts` imports
domain fixtures and returns them for:

- `getExerciseLibrary`
- `getRoutineLibrary`
- `getRoutineDetailView`
- `getDayDetailView`

There are 1,991 lines of fixtures under `src/domain/**/__fixtures__`.

Risk: production bundle includes mock data and "Http" repositories can return
non-HTTP data. This also hides missing platform endpoints behind realistic UI.

Recommendation: either replace these with live endpoints, or split stubs into an
explicit `FixtureTrainingRepository` selected only in a documented dev/test DI
mode. Move large fixtures out of production domain where possible.

### Major - API contract drift guard is partial

Evidence: `contractDrift.test.ts` exists and is useful, but comparison against
real `apiClient.*` calls shows uncovered repository paths in activity,
analytics, measurements, sessions, exercise catalog, and some nutrition/training
surfaces.

Risk: a route can drift or vanish without the local Jest guard failing. The
generated OpenAPI types also exist but are not used by repositories, so most
HTTP calls are handwritten generic casts.

Recommendation: generate the endpoint inventory from repository calls or make
every `apiClient` route declare a contract row. Consider typed endpoint helpers
around `paths`/`components` from `generated/api-types.d.ts` for repo calls.

### Major - Architecture boundaries are not enforced by tooling

Evidence: `eslint.config.js` uses the Expo flat config only. There is no
`no-restricted-imports`, boundary plugin, or architecture-check script in
`package.json`.

Risk: the current architecture can regress without lint/typecheck/test noticing.
The audit scanner found no core inward-dependency violations today, but that is
not protected.

Recommendation: add an architecture check to `npm run lint` or a dedicated
quality script. Enforce at least:

- `domain` cannot import app/application/infrastructure/presentation.
- `application` can import domain only.
- `presentation` can import only domain types and `infrastructure/di/DIContext`.
- `app` can import only screens/app-shell components from presentation.
- `apiClient` can only be imported by infrastructure repositories/tests.

### Major - DI exposes concrete cross-cutting infrastructure services

Evidence: the container exposes `telemetry`, `onboardingDraftStore`,
`languagePreference`, `lastAuthProvider`, and `appInfo` directly. Presentation
uses these through `useDI`, so it avoids direct imports, but it does not always
go through use cases or ports.

Risk: `useDI` becomes a service locator for concrete infrastructure rather than
an application boundary. This is especially easy to expand casually.

Recommendation: formalize cross-cutting ports/use cases for storage,
telemetry, and app metadata, or document a narrow "cross-cutting DI services"
exception with allowed keys.

### Minor - Presentation imports domain runtime values

Evidence: presentation imports values such as `SignInCancelled`,
`ONBOARDING_STEPS`, `HEALTH_METRICS`, `SHOWCASE_METRICS`, and `SHOWCASE_MAX`.
The documented rule says presentation can import domain types only.

Risk: small rule mismatch. These are pure values and may be defensible, but the
rule and code disagree.

Recommendation: either update the rule to allow immutable domain constants and
domain error classes, or move UI-facing lists/errors behind application/presentation
adapters.

### Minor - Open handles after Jest run

Evidence: all tests passed, but Jest reported that it did not exit one second
after completion.

Risk: async resources in tests can create flakiness, hide teardown bugs, or slow
CI.

Recommendation: run `jest --detectOpenHandles` in a follow-up and close the
source, likely around initialized clients/providers.

## Prioritized Next Steps

2. Move `app/_layout.tsx` bootstrap into `presentation/app`.
3. Resolve or isolate `HttpTrainingRepository` fixture-backed methods.
4. Expand contract drift coverage to every real `apiClient` call.
5. Formalize the DI exception for cross-cutting services.
6. Align the "domain types only" rule with current domain constants/error usage.
